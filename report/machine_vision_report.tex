\section{Introduction}
% ============================================================

American Sign Language (ASL) recognition is a high-value assistive technology problem: automated letter-by-letter recognition would enable a cheap camera to serve as a real-time bridge between signing users and text-based interfaces. The goal of this project is to build a complete, real-time ASL gesture recognition system using the Prophesee GenX320 neuromorphic event camera and a lightweight CNN classifier, and to deploy it as a live web dashboard.

Event cameras are a strong fit for this task as an embedded application. Unlike conventional cameras that stream full frames at a fixed rate, the GenX320 reports only per-pixel brightness changes, producing sparse, asynchronous event data with sub-millisecond latency and power consumption under \SI{50}{mW}. This makes event cameras inherently suited for always-on, power-efficient deployment at the edge. The sparse output also reduces the data bandwidth and compute required per inference cycle compared to full-frame video.

The core technical challenge is \textit{cross-sensor domain adaptation}: the only available labelled training data (ASL-DVS) was captured with a different sensor (IniVation DAVIS240C at $120{\times}90$), requiring the model to generalise across different resolutions, aspect ratios, noise profiles, and polarity conventions.

\section{Background Research}
% ============================================================

\subsection{Event Cameras and ASL-DVS}

Event cameras fire asynchronous per-pixel events $e = (t, x, y, p)$ when local log-intensity changes exceed a threshold, providing microsecond temporal resolution and high dynamic range~\cite{lichtsteiner2008}. The Prophesee GenX320 uses Sony's IMX636 stacked sensor with a $320{\times}320$ pixel array, \SI{4.86}{\micro\metre} pixel pitch, and $>\SI{120}{dB}$ dynamic range~\cite{prophesee2023}.

The ASL-DVS dataset~\cite{bi2019} provides 100,800 event recordings of 24 static ASL letters (A--Y, excluding J and Z which require motion) recorded with a DAVIS240C. Prior methods on this dataset used graph-based point-cloud networks achieving ${\sim}91\%$ accuracy. This informs a key design decision: rather than replicating a specialised neuromorphic architecture, we adopt a standard CNN on event-to-frame representations, which is simpler to train and deploy and, as we show, substantially outperforms the prior graph-based approach.

\subsection{Event Representations}

Converting the asynchronous event stream into a tensor suitable for CNN input requires an accumulation scheme. We evaluated three:

\textbf{Two-channel:} Separate ON/OFF count maps $[C_{\text{ON}}, C_{\text{OFF}}]$ per pixel, followed by $\log_{1+x}$ normalisation. Retains full polarity information. \textit{Chosen as primary representation.}

\textbf{Signed:} Single-channel ON$-$OFF difference map. More compact but conflates activity magnitude with polarity sign.

\textbf{Voxel grid:} $B$ temporal bins capturing event order within the window. Richer but slower and showed no improvement for static gestures.

The two-channel representation is the best match for static hand shapes, where spatial polarity structure (which edge of a finger produces ON vs OFF events) is discriminative.

\subsection{Model Architecture}

MobileNetV3-Small~\cite{howard2019} was selected for its favourable accuracy-to-compute ratio on edge hardware. At ${\sim}1.5$M parameters it is significantly smaller than ResNet-18 (${\sim}11$M) or EfficientNet-B0 (${\sim}5.3$M), while achieving competitive accuracy on ImageNet~\cite{deng2009}. Its inverted residual bottlenecks with squeeze-and-excitation modules are well-suited to single-threaded CPU inference, which is our deployment target. Two modifications are made: the first Conv2d is replaced to accept 2-channel event input instead of 3-channel RGB, and the classifier head is repointed to 24 output classes.

\subsection{Cross-Sensor Domain Adaptation}

The GenX320 ($320{\times}320$, 1:1 aspect) and DAVIS240C LR split ($120{\times}90$, 4:3 aspect) differ in resolution, aspect ratio, pixel pitch, and noise floor. Rather than remapping event coordinates before accumulation, we adopt an \textit{accumulate\_then\_resize} strategy: accumulate at native sensor resolution, log-normalise, then bicubic-resize to $256{\times}256$. This makes both training and live inference pipelines converge to the same normalised representation space regardless of source resolution, with no coordinate quantisation artefacts. This choice is motivated by~\cite{bi2019} and the observation that the model only needs to see consistent spatial structure, not absolute pixel coordinates.

\section{Methodology}
% ============================================================

\subsection{Preprocessing and Training Pipeline}

Raw events from a sample or live window are accumulated into a two-channel count map at the native source resolution. Each channel $c \in \{ON, OFF\}$ has pixel value $I_c(x,y) = \sum_i \mathbb{1}[p_i = c, x_i = x, y_i = y]$. Both channels are jointly normalised:
\begin{equation}
    I_{\text{norm}} = \frac{\log(1 + I)}{\log(1 + I_{\max})}
\end{equation}
The resulting $2{\times}H{\times}W$ tensor is bicubic-resized to $2{\times}256{\times}256$. A \texttt{PreprocessConfig} dataclass is embedded in every training checkpoint to guarantee inference alignment.

\subsection{Data Augmentation for Domain Adaptation}

An \texttt{EventFrameAugment} transform is applied stochastically to every training sample to bridge the DAVIS240C$\to$GenX320 domain gap:

\begin{itemize}
    \item \textbf{Vertical flip} (50\%): compensates for opposite y-axis conventions between sensors.
    \item \textbf{Rotation $\pm30^\circ$ + translation $\pm15\%$} (50\%): handles camera angle and hand position variation.
    \item \textbf{Scale 0.7--1.3$\times$ + aspect jitter $\pm15\%$} (50\%): simulates different sensor pixel pitches (\SI{4.86}{\micro\metre} vs \SI{18.5}{\micro\metre}).
    \item \textbf{Contrast scaling 0.6--1.4$\times$} (50\%): simulates different contrast sensitivity.
    \item \textbf{Event dropout} 20\% of pixels (50\%) and \textbf{Gaussian noise} $\sigma{=}0.05$ (50\%): simulate the GenX320's different noise floor and event density.
    \item \textbf{Cutout} 15\% patch (30\%): forces recognition from partial views.
\end{itemize}

\subsection{GenX320 Recorded Data}

To further reduce the domain gap, approximately 1,500 labelled samples were recorded directly on the GenX320 across all 24 classes (${\sim}60$ per class). Samples were collected across varied lighting conditions and hand distances. Because this supplement is 55$\times$ smaller per class than the ASL-DVS training split (60 vs 3,360 samples), a weighted random sampler oversamples GenX320 recordings during training to ensure both sensors contribute meaningfully to each epoch.

\subsection{Training}

Two runs were performed using the hyperparameters in Table~\ref{tab:hyperparams}: \textit{Aug only} (ASL-DVS + augmentation) and \textit{Aug + Recorded} (adding oversampled GenX320 data). Both use AdamW with cosine annealing and label smoothing ($\epsilon{=}0.1$). Training ran on a CPU-only workstation (AMD Ryzen 7 5700X3D, \SI{32}{GB} RAM) without GPU acceleration.

\subsection{Live Inference System}

The deployed system uses the Prophesee Metavision SDK to read \SI{50}{ms} event windows from the GenX320. Each window is converted through the live preprocessing pipeline (source resolution $320{\times}320$, \textit{accumulate\_then\_resize} to $256{\times}256$) and passed to the model. Results are published via a Flask web dashboard displaying the event frame as a red/blue false-colour MJPEG stream, the top prediction with confidence, and per-window latency metrics. The spelling recorder commits a letter only after a configurable streak of consecutive high-confidence windows ($\geq$0.85) for the same class, subject to a per-letter cooldown of \SI{0.8}{s}.

\subsection{Training Performance}

Table~\ref{tab:results} gives test-set accuracy for both training configurations on the 20,160-sample ASL-DVS test set. Both runs converge within 5 epochs to above 99\% validation accuracy and plateau near the ceiling. Adding GenX320 data yields a marginal 0.01 pp gain on the clean DAVIS240C test set; the more meaningful benefit is expected at live deployment where the model has seen the GenX320 sensor's noise and aspect-ratio characteristics.

On plain PyTorch the Pi~5 pipeline measures \SI{47}{ms}, leaving only \SI{3}{ms} of headroom within the \SI{50}{ms} camera window. Exporting the model to ONNX Runtime reduces the pipeline to \SI{39}{ms} (26 FPS), providing comfortable headroom. INT8 post-training quantisation, which the Cortex-A76's dot-product instructions support, could reduce inference further to below \SI{10}{ms}, pushing the full pipeline below \SI{35}{ms}.

\textbf{Power:} The GenX320 sensor draws $<\SI{50}{mW}$ (datasheet~\cite{prophesee2023}). The Pi~5 draws 2.6W at idle and 5--7W under sustained CPU load~\cite{geerling2023}, compared to the development workstation's ${\sim}$40--50W at light load. This 6--8$\times$ power reduction makes the Pi~5 a credible embedded host for battery-powered or wearable deployment.

\subsection{Requirements Evaluation}

The system meets its core goals on the development hardware: 24-class ASL recognition at 60 FPS maximum pipeline throughput with 99.98\% benchmark accuracy. Cross-sensor domain adaptation successfully bridges the DAVIS240C$\to$GenX320 gap without sensor-specific model changes.

On a Raspberry Pi~5 the system achieves real-time operation at 26 FPS with ONNX Runtime export at 5--7W system power. Plain PyTorch on the Pi~5 is marginal (\SI{47}{ms} pipeline, $<$\SI{3}{ms} headroom); ONNX export is therefore required for reliable deployment. The primary gap between the current state and full embedded deployment is not architectural but optimisation: INT8 quantisation of the existing trained model could reduce inference further and provide additional headroom.

% ============================================================
\section{Conclusions}
% ============================================================

A modified MobileNetV3-Small trained on two-channel event frames achieves 99.97--99.98\% accuracy on the ASL-DVS test set, substantially outperforming prior graph-based methods (${\sim}91\%$). The \textit{accumulate\_then\_resize} preprocessing strategy and domain-bridging augmentations make the system deployable on the Prophesee GenX320 without any architectural changes. On a desktop CPU the full pipeline runs at 60 FPS, and on a Raspberry Pi~5 with ONNX Runtime export at 26 FPS, both comfortably supporting real-time inference. The main limitation is the lack of quantitative evaluation on live GenX320 footage; collecting a larger, diverse GenX320 labelled dataset and evaluating INT8 quantisation for further embedded speedup are the clearest next steps.
